# Strategy AI — Отчёт аудита

Проведён аудит проекта: `strategy-ai-full.tsx`, серверные роуты, БД и интеграции. Ниже — найденные проблемы, что улучшить и куда развивать AI.

---

## Критические баги и недочёты

### 1. **Stripe customer_id не подставлялся в req.user** — ИСПРАВЛЕНО
- **Где:** `server/middleware/auth.js` — в SELECT не было `stripe_customer_id`.
- **Что было:** При каждом переходе на платный тариф создавался новый Stripe Customer вместо использования существующего → дубликаты клиентов в Stripe.
- **Исправление:** В выборку пользователя добавлено поле `stripe_customer_id`.

### 2. **WebSocket: синхронизируется только перетаскивание узлов** — ИСПРАВЛЕНО
- **Где:** `strategy-ai-full.tsx` — функции `emitNodeUpdate`, `emitEdgeUpdate` объявлены, но нигде не вызывались.
- **Исправление:** При сохранении узла в NodeEditor вызывается `emitNodeUpdate(fullNode)`. При добавлении/удалении узла — `socket.emit('node-add', …)` / `socket.emit('node-delete', …)`. Введён `setEdgesUser(updater)` и вызовы `emitEdgeUpdate` при любом пользовательском изменении рёбер (связи, шаблоны, импорт, восстановление версии).

### 3. **Зависимость WebSocket от `(window as any).io`** — ИСПРАВЛЕНО
- **Исправление:** В `package.json` добавлена зависимость `socket.io-client`, в `strategy-ai-full.tsx` — `import { io as ioClient } from 'socket.io-client'` и подключение через `ioClient(API_BASE, …)`.

### 4. **Dev-аккаунт захардкожен в двух местах** — ИСПРАВЛЕНО
- **Исправление:** На бэкенде `DEV_EMAILS` берётся только из `process.env.DEV_EMAILS` (без дефолтного email); в ответах login/register/profile добавлено поле `is_dev`. На фронте проверки заменены на `user.is_dev` вместо хардкода email. В офлайн-режиме (без API) по-прежнему используется seed в localStorage для демо.

### 5. **Смена пароля в режиме без API (localStorage)**
- **Где:** `ProfileModal` → `changePw()` — при `!API_BASE` пароль «меняется» через `patchUser(..., { pwHash })`, но на бэкенде нет PATCH для пароля по hash; в localStorage хранится `pwHash: hashPw(email, pw)`.
- **Что не так:** `patchUser` в режиме без API пишет в `sa_acc` в store; при этом хэш пароля не хэшируется через bcrypt — используется `hashPw` (btoa). Это небезопасно и несовместимо с бэкендом (bcrypt). Для офлайн-режима можно оставить как есть с пометкой «только для демо», но не считать смену пароля безопасной.

### 6. **Несогласованность лимитов тарифов (фронт vs бэкенд)** — ЧАСТИЧНО ИСПРАВЛЕНО
- **Исправление:** В `server/routes/tiers.js` лимиты сценариев приведены в соответствие с фронтом: starter 2, pro 5, team 15. Единый источник правды (фронт тянет с API) можно добавить позже.

### 7. **Ошибка 500 при сохранении карты: invalid UUID** — ИСПРАВЛЕНО
- **Где:** Клиент создаёт карты с `id: uid()` (короткий id вроде `k@vepbq`), при сохранении шёл PUT на `/api/projects/.../maps/k@vepbq`, бэкенд ожидает UUID.
- **Исправление:** В `saveMap` добавлена проверка `isUUID(map.id)`. Если id не UUID — выполняется POST (создание), возвращается карта с серверным UUID. Все вызовы (createMapWithTemplate, createBlankScenario, createScenarioFromTemplate, doSaveAndGo) используют возвращённую карту для открытия/обновления состояния.

### 8. **Цветовые темы и палитра не применялись** — ИСПРАВЛЕНО
- **Исправление:** В `public/index.html` добавлены критические стили `body[data-theme="dark"]` и `body[data-theme="light"]` (фон и цвет текста), чтобы тема применялась до загрузки React. Скрипт установки `data-theme`/`data-palette` из localStorage уже был; теперь визуал совпадает с выбранной темой с первого кадра.

### 9. **Стеклянный дизайн (glass) не был виден** — ИСПРАВЛЕНО
- **Где:** У модалок с классом `glass-panel` в inline-стиле стоял `background:"var(--bg2)"`, перекрывавший полупрозрачный фон из CSS.
- **Исправление:** С инлайн-стилей всех панелей с классом `glass-panel` убран `background`; используется только фон из класса (blur + полупрозрачность).

### 10. **API-ключ OpenAI**
- **Где:** Сервер использует `process.env.OPENAI_KEY` (`server/routes/ai.js`), клиент при офлайне — `window.__STRATEGY_AI_OPENAI_KEY__`.
- **Что проверить:** В Railway (или в окружении деплоя) в Variables должна быть переменная **OPENAI_KEY** (значение `sk-...`). Без неё AI на сервере не работает; сообщение об ошибке на клиенте указывает на необходимость настройки ключа.

## Логика настроена не до конца

### 11. **AI на тарифе Free** — ИСПРАВЛЕНО
- **Исправление:** В `AiPanel` при `tier === 'free'` поле ввода и кнопка отправки отключены, показывается заглушка «AI-чат доступен с тарифа Starter». При попытке отправить сообщение добавляется системное сообщение с предложением улучшить тариф; запрос к API не отправляется.

### 12. **Экспорт PDF / PPTX**
- **PDF:** Реализован через `window.open` + HTML-таблица + `window.print()` → пользователь сохраняет как PDF через диалог печати. Это ок для MVP, но в лендинге/фичах можно уточнить: «Экспорт в PDF через печать браузера».
- **PPTX:** Аналогично — открывается окно с HTML-«слайдами» и `window.print()`. Реального файла .pptx не создаётся; в Enterprise заявлено «PPTX-отчёты для совета директоров» — лучше либо реализовать генерацию .pptx (библиотека вроде pptxgenjs), либо переформулировать в «подготовка к экспорту в PPTX (печать по слайдам)».

### 13. **Поделиться картой (share)**
- Есть кнопка и копирование ссылки; при наличии API данные сохраняются через бэкенд (shares). Для режима без API ссылка с `#share=...` и сохранение в localStorage — в RECOMMENDATIONS уже отмечено, что кросс-устройство не работает без бэкенда. Логика «только просмотр по ссылке» на фронте реализована.

### 14. **Cron напоминаний о дедлайнах** — ИСПРАВЛЕНО
- **Исправление:** Введена переменная `lastDeadlineRunDate`; напоминания запускаются только при `hour === 8 && minute < 5` и только если за текущий день ещё не запускались (`lastDeadlineRunDate !== today`).

## Что можно улучшить / добавить в систему

1. **Единый конфиг тарифов** — ИСПРАВЛЕНО  
   Создан `shared/tiers.json` — единый источник правды. Бэкенд читает из него при старте; `server/routes/tiers.js` и `server/routes/maps.js` используют TIERS для лимитов. Stripe price_id подставляется из env.

2. **Офлайн / ретраи при старте**  
   При первом загрузке обернуть `getSession()` / загрузку проектов в try/catch и показывать экран «Не удалось загрузить данные — повторить» вместо белого экрана или необработанного исключения.

3. **Валидация и доступ по ролям**  
   - Проверка прав на бэкенде при каждом изменении карты/проекта (уже есть `getProjectAccess`, `role === 'viewer'`).  
   - На фронте для роли «viewer» скрывать кнопки редактирования и не давать вызывать API изменения.

4. **Лимит сценариев по тарифу** — ИСПРАВЛЕНО  
   Добавлена проверка лимита сценариев при создании карты с `is_scenario = true`. Бэкенд `GET /api/tiers/usage` возвращает `scenarios.used` и `scenarios.limit`. В `maps.js` при POST проверяется лимит сценариев по тарифу.

5. **История версий карты** — ДОПОЛНЕНО  
   Добавлена кнопка «📸 Версия» в тулбаре редактора — ручное сохранение снимка. Автосохранение при PUT (каждые 10 мин) уже было. Полный цикл: сохранение версии, список, восстановление — работает.

6. **Уведомления (in-app)**  
   Есть таблица `notifications` и создание записей из webhooks. Нужен UI: иконка колокольчика, список непрочитанных, пометка прочитанным и переход по `link`.

7. **Поиск по картам/проектам**  
   Есть роут поиска; подсветка результатов и переход к шагу на карте улучшат UX.

8. **Экспорт в настоящий PPTX** — ИСПРАВЛЕНО  
   Для Enterprise (tier.pptx) добавлена генерация .pptx через pptxgenjs: титульный слайд + слайды по каждому шагу (название, статус, приоритет, описание, метрика, прогресс). Для остальных тарифов — печать по слайдам (HTML).

9. **Тесты**  
   Хотя бы smoke: логин → создание проекта → создание карты → вызов AI (мок) → сохранение. Плюс тесты для лимитов тарифов и webhook Stripe.

10. **Мониторинг** — ДОПОЛНЕНО  
    В `server/routes/ai.js` добавлено логирование ошибок AI: `[AI] Error`, `[AI] OpenAI error` — без тела запроса, с tier и маскированным email. Sentry уже подключён по SENTRY_DSN.

## Как улучшить AI и в каком направлении развивать

### Текущее состояние
- Есть тарифные уровни (free → enterprise) с разной длиной/глубиной системных промптов и базой знаний (AI_KNOWLEDGE).
- Контекст: проект, карта, шаги, связи, статистика. Есть распознавание намерений и формат `<ADD>{...}</ADD>` для предложения шагов.
- Бэкенд: лимиты сообщений, модели gpt-4o-mini / gpt-4o по тарифу, один эндпоинт `/api/ai/chat`.

### Направления развития

1. **Контекст и память**
   - Хранить сжатое резюме диалога (например, последние N сообщений + summary) в БД или в состоянии карты, чтобы при новом заходе AI помнил предыдущие выводы и не повторял одни и те же вопросы.
   - Добавить в контекст последние действия пользователя: «только что добавил шаг X», «сменил статус Y на completed» — для более релевантных советов.

2. **Специализированные сценарии (проактивный AI)**
   - «Аудит карты» — по кнопке один раз проходится по всей карте и выдаёт отчёт: разрывы, блокировки, слабые метрики, рекомендации.
   - «Следующий шаг» — не только по запросу в чате, но и автоматическая подсказка при открытии карты: «Сейчас логично сделать X, потому что Y».
   - «Риски» — отдельный режим: AI перечисляет риски по текущим шагам и связям и предлагает митигации.

3. **Качество ответов и формат**
   - Чёткие инструкции по языку ответа (из `ai_lang` пользователя) и по длине (кратко / развёрнуто) в зависимости от тарифа.
   - Структурированный вывод для Pro+: блоки «Диагноз», «Рекомендации», «Риск», «Быстрая победа» — парсить и рендерить как блоки в UI, а не только сплошной текст.

4. **Предложение связей и шагов**
   - Уже есть «AI-связи» (autoConnect). Можно расширить: после добавления нового узла AI предлагает 1–2 варианта связей с существующими («связать с шагом X по типу requires»).
   - При пустой карте после интервью — проверка качества сгенерированной карты (нет ли дублей, все ли связи осмысленны) и одна итерация «улучшить формулировки узлов».

5. **Мультимодальность и интеграции**
   - Загрузка файла (скриншот схемы, PDF стратегии) и вопрос по нему: «Вот наша текущая стратегия — что бы ты изменил?» (требует vision API или отдельный пайплайн).
   - Для Enterprise — «C-level коллегиум»: несколько виртуальных ролей (CSO, CMO, CFO), каждый отвечает с своей точки зрения; можно оформить как отдельный режим чата с выбором роли.

6. **Метрики и обучение**
   - Логировать (без персональных данных): тариф, длина контекста, время ответа, был ли использован быстрый ответ (thumbs up/down). Анализировать, на каких типах запросов отказы или нерелевантные ответы — донастраивать промпты и примеры.

7. **Безопасность и лимиты**
   - Уже есть лимит сообщений в месяц и обрезка контекста. Имеет смысл лимитировать длину одного сообщения пользователя и суммарный размер контекста в токенах (оценка или через API), чтобы не упираться в лимиты модели и не раздувать стоимость.

Итог по AI: сильная база (тарифы, контекст карты, база знаний). Главные шаги вперёд — более богатый контекст и память, проактивные сценарии (аудит, следующий шаг, риски), структурированный вывод в UI и опционально — мультимодальность и мультиролевой режим для Enterprise.
